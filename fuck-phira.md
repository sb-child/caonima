# fuck-phira 详细文档

除了为 phira 改采样率，和像 [fuck-pipewire-pulse](./fuck-pipewire-pulse.py) 那样叫醒 pipewire 之外...

本程序的另一个用途是，在没有输出设备(Audio Sink)可用时，通过[配置文件](#配置文件文档)规则匹配进程并发送信号(signal)。

## 配置文件文档

配置文件基于 JSON 格式编写，由一个包含多个“选择器对象”的数组（List）组成。每个选择器独立运作。

配置文件需要存放在 `~/.caonima/fuck-phira/kill_program_on_sink_unavailable.json`。

文件中可以定义进程选择器(Selectors)的规则集合，程序会依次执行每一条规则。结构和字段描述见下文。

### 基础结构

配置文件的根节点必须是一个 **JSON 数组** (`[]`)。数组中的每一个元素都是一个 **JSON 对象** (`{}`)，代表一条独立的匹配与执行规则。

```json
[
  {
    "desc": "规则描述 1",
    "name": "process_name",
    "fullpath": "/absolute/path/to/executable",
    "signal": 9
  },
  {
    "desc": "规则描述 2",
    ...
  }
]

```

### 字段说明

每个选择器对象支持以下核心字段：

| 字段名     | 数据类型        | 必填 | 默认值             | 说明                                                                                                 |
| ---------- | --------------- | ---- | ------------------ | ---------------------------------------------------------------------------------------------------- |
| `desc`     | String          | 否   | `selector_{index}` | **选择器的名称或描述**: 规则标识，在数组中保持唯一性。                                               |
| `name`     | String / `null` | 是   | 无                 | **目标进程的简短名称**: 相当于 Linux 下的 `comm`。如果不通过进程名匹配，请显式设置为 `null`。        |
| `fullpath` | String / `null` | 是   | 无                 | **目标程序的绝对路径**: 相当于 Linux 下的 `/proc/PID/exe`。如果不通过路径匹配，请显式设置为 `null`。 |
| `signal`   | Integer         | 是   | 无                 | **需要发送的系统信号编号**: 当进程被成功匹配时，系统将向该进程发送此信号。                           |

### 匹配逻辑与生效规则

程序的匹配引擎会严格按照 `name` 和 `fullpath` 字段的值进行判定。逻辑如下：

1. **精确名称匹配** (`name`)：

- 如果配置了具体的字符串（如 `"nginx"`），则只有进程名完全一致的进程才会命中。

2. **精确路径匹配** (`fullpath`)：

- 如果配置了具体的路径（如 `"/usr/local/nginx/sbin/nginx"`），则只有从该绝对路径启动的进程才会命中。

3. **组合匹配** (AND 逻辑)：

- 如果 `name` 和 `fullpath` **都不为** `null`，则目标进程必须**同时满足**这两个条件才会触发信号发送。这适用于系统中存在同名进程，但需要精确定位特定目录下程序的情况。

4. **规则失效/禁用**：

- 如果 `name` 和 `fullpath` **均为** `null`，则该选择器将被视为未激活（失效）状态。程序会跳过此规则，不会对任何进程进行匹配或操作。这是一种快速禁用某条规则的便捷方法。

### 常用系统信号参考 (`signal` 字段)

在 Linux 系统中，常用的信号枚举值如下：

- **`9` (SIGKILL)**：无条件强制终止进程。进程无法捕获或忽略此信号，是最常用的“强杀”手段。
- **`15` (SIGTERM)**：请求程序正常终止。程序可以捕获此信号并执行清理工作（如保存数据、关闭文件）后再退出，推荐优先使用。
- **`2` (SIGINT)**：终端中断信号（相当于在终端按下 `Ctrl+C`）。
- **`1` (SIGHUP)**：终端断线信号。常用于通知守护进程重新读取配置文件（如 Nginx、服务重启）。

### 完整配置示例

以下是一个包含了各种匹配场景的完整配置文件示例：

```json
[
  {
    "desc": "强杀挖矿木马程序 (仅通过名称匹配)",
    "name": "xmrig",
    "fullpath": null,
    "signal": 9
  },
  {
    "desc": "平滑重启特定目录的 Nginx (名称+路径双重匹配)",
    "name": "nginx",
    "fullpath": "/opt/custom_nginx/sbin/nginx",
    "signal": 1
  },
  {
    "desc": "清理旧版测试工具 (仅通过路径匹配)",
    "name": null,
    "fullpath": "/home/user/workspace/test_app/bin/legacy_tool",
    "signal": 15
  },
  {
    "desc": "这是一个被禁用的规则，不会产生任何效果",
    "name": null,
    "fullpath": null,
    "signal": 9
  }
]
```
